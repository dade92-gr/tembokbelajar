Perancangan Database dan Arsitektur Aplikasi Web

Semester Genap - Kelas X (Proyek Tahap 3) 4 JP

A. Pendahuluan

Setelah kita merumuskan solusi dan menentukan fitur-fitur yang akan dikembangkan, langkah selanjutnya adalah merancang bagaimana data akan disimpan dan bagaimana aplikasi akan bekerja. Tahap ini sangat penting karena keputusan yang diambil akan mempengaruhi kemudahan pengembangan, kinerja aplikasi, dan kemudahan pemeliharaan di masa depan.

Bayangkan kamu akan membangun sebuah rumah. Sebelum membangun, kamu perlu membuat denah (rancangan ruangan) dan struktur bangunan (rangka, pondasi). Tanpa denah yang jelas, rumah bisa jadi tidak teratur dan sulit ditempati. Tanpa struktur yang kuat, rumah bisa ambruk. Hal yang sama berlaku dalam pengembangan aplikasi.

B. Apa Itu Perancangan Database?

Perancangan database adalah proses menentukan bagaimana data akan disimpan, diorganisir, dan dihubungkan satu sama lain dalam basis data. Database yang dirancang dengan baik akan membuat aplikasi cepat dalam mengakses data, akurat karena data terstruktur dengan baik, mudah dikembangkan, dan mudah dipelihara.

C. Entity Relationship Diagram (ERD)

ERD (Entity Relationship Diagram) adalah diagram yang menggambarkan hubungan antar entitas (tabel) dalam basis data. ERD adalah "peta" yang menunjukkan bagaimana data saling berhubungan.

Contoh ERD Sederhana:
┌─────────────────┐         ┌─────────────────┐
│     SISWA       │         │     NILAI       │
├─────────────────┤         ├─────────────────┤
│ NIS (PK)        │───┐     │ ID_NILAI (PK)   │
│ Nama            │   │     │ NIS (FK)        │
│ Kelas           │   └────►│ MataPelajaran   │
│ Alamat          │         │ Nilai           │
└─────────────────┘         └─────────────────┘

PK = Primary Key (Kunci Utama)
FK = Foreign Key (Kunci Asing)
───► = Relasi One to Many (Satu Siswa memiliki banyak Nilai)

D. Jenis-jenis Relasi

Klik pada setiap jenis relasi untuk memperjelas tampilan:

One to One (1:1)
Satu record di tabel A berhubungan dengan satu record di tabel B. Contoh: Satu siswa memiliki satu kartu pelajar.
One to Many (1:N)
Satu record di tabel A dapat berhubungan dengan banyak record di tabel B. Contoh: Satu guru mengajar banyak mata pelajaran.
Many to Many (N:M)
Banyak record di tabel A berhubungan dengan banyak record di tabel B. Contoh: Banyak siswa mengambil banyak mata pelajaran (membutuhkan tabel penghubung).

E. Langkah-langkah Perancangan Database

Klik pada setiap langkah untuk memperjelas tampilan:

1
Identifikasi Entitas
Tentukan objek/konsep yang datanya perlu disimpan. Lihat fitur yang akan dikembangkan.
2
Tentukan Atribut
Tentukan detail informasi yang perlu disimpan dari setiap entitas.
3
Tentukan Primary Key
Tentukan identitas unik setiap record (PK).
4
Tentukan Relasi
Tentukan hubungan antar entitas (1:1, 1:N, N:M).
5
Tentukan Foreign Key
Tentukan atribut penghubung antar tabel (FK).

F. Tipe Data dalam Database

Klik pada setiap tipe data untuk memperjelas tampilan:

INT
Bilangan bulat untuk ID, jumlah, stok, umur
VARCHAR(n)
Teks dengan panjang maksimal untuk nama, alamat, judul
TEXT
Teks panjang untuk deskripsi, artikel
DATE
Tanggal (YYYY-MM-DD) untuk tanggal lahir, transaksi
DECIMAL(p,s)
Bilangan desimal untuk harga, nilai ujian
BOOLEAN
Nilai benar/salah untuk status aktif, sudah lunas

G. Normalisasi Database

Bentuk Normal (Normal Form):
• 1NF - Setiap kolom berisi nilai atomic (tidak boleh ada kolom dengan lebih dari satu nilai)
• 2NF - Tidak ada ketergantungan parsial (setiap kolom non-key bergantung pada seluruh PK)
• 3NF - Tidak ada ketergantungan transitif (kolom non-key tidak bergantung pada kolom non-key lain)
Contoh Normalisasi: SEBELUM (Pelanggaran 1NF): ID_Siswa | Nama | Telepon 001 | Andi | 081234, 085678 SETELAH NORMALISASI 1NF: ID_Siswa | Nama | Telepon 001 | Andi | 081234 001 | Andi | 085678 SETELAH NORMALISASI 2NF & 3NF: Tabel SISWA: Tabel TELEPON: ID_Siswa | Nama ID_Telepon | ID_Siswa | Nomor_Telepon 001 | Andi 1 | 001 | 081234 2 | 001 | 085678

H. Arsitektur Aplikasi Web

Arsitektur 3-Tier (Lapis 3) adalah struktur yang paling umum untuk aplikasi web.

Presentation Tier
Frontend - Tampilan dan interaksi pengguna (HTML, CSS, JS)
Logic Tier
Backend - Proses data, logika bisnis, validasi (PHP, Python, Node.js)
Data Tier
Database - Penyimpanan dan pengambilan data (MySQL, PostgreSQL)
Alur Data dalam Aplikasi Web:
PENGGUNA
    │
    ▼
1. Pengguna membuka browser dan mengakses URL
    │
    ▼
2. Browser mengirim request ke Web Server
    │
    ▼
3. Web Server meneruskan request ke Backend
    │
    ▼
4. Backend memproses request (validasi, logika)
    │
    ▼
5. Jika perlu data, Backend mengambil dari Database
    │
    ▼
6. Database mengembalikan data ke Backend
    │
    ▼
7. Backend mengolah data dan menghasilkan response
    │
    ▼
8. Response dikirim ke browser (HTML/JSON)
    │
    ▼
9. Browser menampilkan halaman kepada pengguna

I. Memilih Teknologi yang Tepat

📌 Rekomendasi Teknologi untuk Pemula:
• Frontend: HTML, CSS, JavaScript (dasar, mudah dipelajari)
• Backend: PHP (mudah, banyak hosting murah) atau Python (rapi, mudah dibaca)
• Database: MySQL (populer, mudah, banyak dokumentasi)

J. Etika dalam Perancangan Database

Prinsip Etika:
• Privasi: Hanya simpan data yang benar-benar diperlukan
• Keamanan: Password harus dienkripsi, bukan disimpan plain text
• Transparansi: Pengguna harus tahu data apa yang dikumpulkan
• Hak Pengguna: Rancang mekanisme untuk melihat, memperbaiki, atau menghapus data

K. Rangkuman

  • Perancangan database adalah proses menentukan bagaimana data akan disimpan dan diorganisir.
  • ERD (Entity Relationship Diagram) adalah diagram yang menggambarkan hubungan antar entitas.
  • Jenis relasi: One to One (1:1), One to Many (1:N), Many to Many (N:M).
  • Primary Key adalah identitas unik setiap record; Foreign Key adalah penghubung antar tabel.
  • Normalisasi adalah proses mengurangi redundansi data (1NF, 2NF, 3NF).
  • Arsitektur 3-Tier: Presentation Tier (Frontend), Logic Tier (Backend), Data Tier (Database).
  • Etika dalam perancangan database: privasi, keamanan, transparansi, hak pengguna.
Ingat! Perancangan database dan arsitektur yang baik adalah fondasi dari aplikasi yang handal. Luangkan waktu untuk merancang dengan matang, karena perubahan struktur database di tengah pengembangan akan sangat merepotkan!
